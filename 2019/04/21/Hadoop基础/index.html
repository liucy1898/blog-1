3. Hadoop 配置

Hadoop 的配置都是 XML 文件的方式完成, 通用配置都在 core-site.xml 中, HDFS, MapReduce 和 YARN 都有对应的 hdfs-site.xml, mapred-site.xml 以及 yarn-site.xml.

Hadoop 的设计的目的在于处理海量数据, 其主要内容包括数据的存储以及运算, 存储使用 HDFS 实现, 运算使用 MapReduce 编程模型实现.

Hadoop 有三种配置方式:

  1. 独立模式: 没有守护程序, 所有程序都运行在一个单独的 JVM 之上, 独立模式适合在开发期间运行 MapReduce 程序, 方便调试和测试.
  2. 伪分布式: Hadoop 守护程序运行在本地机器上, 会模拟一个小规模的集群.
  3. 完全分布式: 运行在集群的不同机器上.

当需要运行某个模式的 Hadoop 时, 需要设置适当的配置, 以及启动守护进程(独立模式除外), 不同模式见的配置如下:

配置文件 属性 独立模式值 伪分布式值 完全分布式值
core fs.defaultFS file:///(默认值) hdfs://localhost/ hdfs://namenode
HDFS dfs.replication N/A 1 3(默认值)
MapReduce mapreduce.framework.name local(默认值) yarn yarn
yarn yarn.resourcemanager.hostname
yarn.nodemanager.aux-services		 N/A
N/A		 localhost
mapreduce_shuffle		 resourcemanager
mapreduce_shuffle

此外, Hadoop 的不同配置模式见可以共存, 只需用不同的目录存放配置文件即可, 启动的时候可以通过如下两种方式来指定配置文件:

  1. 设置 HADOOP_CONF_DIR 环境变量
  2. 通过 --config 选项来指定

3.1 独立模式

在独立模式下不需要进行额外的配置, 所有默认的属性都是针对独立模式的, 也没有守护程序运行, 独立模式下使用的文件系统是 Local File System 和 Local MR job runner.

可以看到, 独立模式下使用 hadoop fs -ls 显示的就是本机的根路径文件

在伪分布式下, 必须要启动守护进程, 启动守护进程就需要使用提供的启动脚本, Hadoop 并不严格区分伪分布式和完全分布式, 只是在目标主机上启动守护进程, 通过 SSH 命令让主机之间相互通信, 而且要启动守护进程.

伪分布式只是完全分布式的一个特例, 是一个在单个主机上运行 Hadoop 完全分布式的场景, 因此我们需要确保能够通过 SSH 命令登录本机, 而不需要通过输入密码.

4. 架构分析

4.1 HDFS 架构分析

Hadoop 分布式文件系统

  • 按需定制 MapReduce(Map + Reduce 处理大量纯文本)
  • 目标在于多次的文件流读取(适用于一次写入多次读取)
  • 写入成本高
  • 高度数据冗余(副本数量默认为 3)
  • 每个节点不需要 RAID(独立磁盘冗余阵列, 将读写分散在多个硬盘中, 提高吞吐量. 常见等级有 RAID-0, RAID-1, RAID-5, RAID-10)
  • BlockSize 比较大(默认 128M)
  • 定制节点的位置感知

对 Hadoop 文件系统来说, 首先会将文件切割成 128M 的 BlockSize, 然后再将每个 Block 冗余三份, 存放在不同的 DataNode 之上.

4.1.1 特点

  1. Hadoop 能够存储超大规格的文件(PB 级别)
  2. 一次写入, 多次读取的模式是最高效的, 一个数据集通常由数据源生成或复制.
  3. Hadoop 不需要运行在昂贵并且高可靠的机器上, 普通的商用硬件即可满足需求.

同时, Hadoop 依然存在不适用的场景:

  1. 低延迟数据访问: 往往 HDFS 无法满足毫秒级的数据访问延迟, HDFS 主要是为高吞吐量而设计和优化. 低延迟的访问场景下, 往往 HBase 是更好的选择.
  2. 大量小文件: NameNode 存储着文件系统的元数据, 因此文件数量的限制也由 NameNode 内存量决定. 每个文件的索引目录通常占 150 个字节.
  3. 随机写: HDFS 中的文件只有一个写入者, 并且写操作总是作用在文件的末尾(追加).

4.2 NameNode

客户端通过 NameNode 和 DataNode 交互来访问整个文件系统, 客户端提供一个类似 POSIX(可移植操作系统界面) 的文件系统接口, 用户在编程时不需要知道名称节点和数据节点及其功能.

NameNode 管理文件系统的命名空间, 维护着这个文件系统数内所有的文件和索引信息. 这些信息以命名空间镜像和编辑日志的形式永久保存在磁盘. NameNode 也记录着每个文件的每个块所在的数据节点, 但它并不永久保存块的位置, 因为这些信息会在系统启动的时候有 DataNode 重建.

  • 存储文件元数据, 比如目录结构
  • 运行 NameNode 的服务器至关重要, 只有 1 个
  • 只对元数据的增删改查做日志记录, 不对 Block 和文件流做记录
  • DataNode 故障时, 负责创建更多的副本 Block

4.3 DataNode

数据节点时文件系统的工作者, 它们存储并提供定位块的服务, 并且定时向NameNode 发送他们存储的快的列表.
