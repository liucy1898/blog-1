如何设计权限管理系统

发表于 2019-10-18

1 权限管理核心

1.1 用户-权限模型

直接对用户完成权限的管理, 相对比较简单的权限场景, 适合权限是完全固定, 并且分配的人员数量也不多的情况. 类似 MySQL 就是通过用户完成权限分配的.

1.2 RBAC(Role-Based-Access-Control, 用户-角色-权限)模型

基于角色的权限访问控制. 在 RBAC 中, 权限与角色相关联, 用户通过成为不同的角色从而得到相应的权限.

RBAC 关注的核心是 User, Role 和 Permission 三者间的关系, 用户和角色, 角色和权限之间都是多对多关系, 此外, RBAC 的设计遵循以下三个原则:

  1. 最小权限原则: RBAC 可以将角色配置为
  2. 责任分离原则: RBAC 可以
  3. 数据抽象原则

相关的操作界面:

  1. 基础数据的配置界面:
    1. 权限管理界面: 会引入权限模块的概念, 将权限按模块划分开方便管理. 同时支持对权限模块和权限的增删改查.
    2. 角色管理界面: 做角色的增删改查.
    3. 用户管理界面: 除了对用户的增删改查外, 还可以引入 Group 的概念, 以及 Group 的增删改查.
  2. 权限关系的维护界面:
    1. 角色和权限关系维护界面
    2. 用户和角色关系维护界面

2. Spring Security

2.1 简介

Spring Security 提供了大量的过滤器, 拦截 Servlet 请求, 并将这些请求转交给认证处理过滤器和访问决策过滤器进行处理, 并强制性的认证用户身份和权限, 以达到保护 web 资源的能力.

  • 认证: 是为用户建立一个他所声明的主体的过程, 这个主体一般是用户, 设备或者其他可执行具体行为的实体.
  • 验证: 指用户能否在应用中执行某个操作.

常见的验证模式:

名称 验证方式 优点 缺点
Basic 认证 基于特定的资源需要提供用户名和密码后才可访问, 需要认证时, 浏览器会先弹出输入框, 当用户输入用户名密码后, 浏览器使用 用户名:密码 的格式完成 base64 编码, 并放在 Http 请求的 header 部分, 服务器解析信息并通过后才能将进行后续的操作 1. HTTP 是无状态协议导致每次请求都要带上认证信息
2. 传输安全性较差
Digest 认证 主要为了解决 basic 模式的安全问题, 用于替代 Basic. 浏览器依旧会弹出输入框, 浏览器会对用户名, 密码, HTTP 请求方法, URI 等进行 MD5 加密, 服务器获取后从中获取用户名, 同样对用户名, 密码, HTTP 请求方法, URL 再次进行 MD5 加密, 如果和请求中的 MD5 值相同则认为认证成功. 避免了直接传输密码, 提高安全性 依然存在安全问题, 认证的报文如果被拦截, 攻击者可以原样发送给服务器
X.509 证书认证 是种通用的证书格式, 包括证书版本, 持有者的公钥, 证书序列号. 序列号是由 CA 签发的每个分配者唯一的编号, 此外还包括主体信息, 证书有效期, 认证机构, 发布者数字签名, 签名算法等.
LDAP 认证

2.2.1 权限拦截器

过滤器 功能
SecurityContextPersistenceFilter 位于过滤器链的顶端, 是第一个生效的过滤器, 执行其他过滤器之前率先判断用户的 session 是否已经存在于 security context, 如果存在就放在 SecurityContextHolder 中供其他过滤器使用, 否则创建. 并且当所有过滤器执行完毕后清空 SecurityContextHolder 的内容.
LogoutFilter 只处理注销请求, 在用户发送注销请求时, 清空用户 session 和 SecurityContextHolder, 并重定向到注销成功页面
AbstractAuthenticationProcessingFilter 处理表单登录的过滤器, 通过用户名和密码判断接收的请求是否有效
DefaultLoginPageGeneratingFilter 生成默认的登录页面
BasicAuthenticationFilter 进行 Basic 认证
SecurityContextHolderAwareRequestFilter
RememberMeAuthencationFilter 实现 RememberMe 功能, 依赖于 cookie
AnonymousAuthenticationFilter 保证操作统一性, 当用户没有登录时, 默认分配匿名用户的权限
ExceptionTranslationFilter 处理 SpringSecurity 抛出异常, 并重定向到对应页面或返回对应的错误代码
SessionManagementFilter 为了防御 session 伪造攻击, 在登录成功后销毁当前 session 并重新生成
FilterSecurityInterceptor 用户的权限控制都包含在这个拦截器中

2.2.2 数据库管理

  1. 当用户登录时, 需要先执行身份认证
  2. 认证通过后, 调用角色管理器判断是否有权限访问
  3. 需要依赖于用户表, 角色表, 权限表以及用户-角色关联表, 角色-权限关联表

2.2.3 权限缓存

SpringSecurity 实现了缓存 UserDetailsService 的实现类 CachingUserDetailsService

2.2.4 自定义决策

当用户身份认证通过后, 会调用决策管理器决定是否能完成此次访问, 该行为是由 AccessDecisionManage 接口控制的, 我们自定义该决策的时候一般可以继承 AbstractAccessDecisionManage 抽象类即可. SpringSecurity 在这里会使用一个投票器, 通过循环判断用户是否有访问资源所需要的权限, 最终决定是否允许访问.

目前 SpringSecurity 实现的投票器:

  • AffirmativeBased: 一票通过
  • ConsensusBased: 半数通过
  • UnanimousBased: 全票通过

2.3 优缺点总结

优点:

  • 提供了一套安全框架
  • 提供了很多用户认证的功能, 实现相关接口即可, 节约大量开发工作
  • 基于 Spring, 易于集成到 Spring 项目中, 且封装了很多方法

缺点:

  • 配置文件多, 角色被编码到配置文件和源码中, RBAC 不明显
  • 对于系统中用户, 角色, 权限之间的关系没有可操作的界面
  • 对于大数据量不友好

3. Shiro

核心关注点:

  • Authentication: 用户识别
  • Authorization: 访问控制
  • Session Management:
  • Cryptography: 对数据源加密的同时保证容易使用

其他支持:

  • Web Support: 能够快速集成 web 应用
  • Caching: 保证程序能够高效获取权限信息
  • Concurrency: 支持多线程并发
  • Test: 支持单元测试集成测试
  • Run As: 允许用户设定其他
  • Remember Me: 跨 session 记录用户的身份, 只有强制需要时才会登录
